# Introdução ao Git ----- ## O que é versionamento centralizado? ![Diagrama de um versionamento centralizado](https://git-scm.com/book/en/v2/images/local.png) > * servidor contém todas as versões > * usuários (nós) baixamos a última versão > * lembra vagamente a ideia de criar pastas com datas ----- ## O que é versionamento distribuído? ![Diagrama de um versionamento distribuído](https://git-scm.com/book/en/v2/images/distributed.png) > * servidor contém todas as versões > * usuários (nós) também temos todas as versões > * pensado para trabalhar distribuído/sem internet sempre ----- ## Instalando * Download a partir de https://git-scm.com/downloads * funciona em Windows, Mac e Linux ----- ## Instalando Para testar, digite no seu terminal e tecle enter: git --version Alguma versão maior que 2.0 deve aparecer, como por exemplo `git version 2.14.1` ----- ## Configurando meu usuário * você precisa se identificar para o mundo * o e-mail é o meio utilizado ----- ## Configurando meu usuário git config --global user.name Mauricio Borges Silva git config --global user.mail mbsilva@thoughtworks.com ----- ## Criando meu primeiro projeto * todo projeto criado é um repositório * um repositório é, em suma, o conjunto de alterações efetuadas no conteúdo de uma pasta (histórico) * o git consegue saber como lidar com esse fato, mesmo que pessoas trabalhem em paralelo nos mesmos arquivos ----- ## Criando meu primeiro projeto git init meu-primeiro-projeto # cria uma pasta configurada para lidar com git cd meu-primeiro-projeto # navega para a pasta ls -a # lista arquivos ocultos ----- ## Versionando meus arquivos touch index.html # cria um arquivo vazio atom index.html # edita o arquivo criado com o Atom, coloca um hello world :p ----- ## Versionando meus arquivos git status # exibe o status do repositório atual ----- ## Versionando meus arquivos git add index.html # exibe o status do repositório atual ----- ## Versionando meus arquivos ![Os diferentes estágios de um arquivo](https://git-scm.com/book/en/v2/images/areas.png) > * commited: está salvo na base de dados local (em resumo, dentro da pasta .git num formato que a ferramenta entende) > * modified: o arquivo foi alterado > * staged: o arquivo foi alterado, e o git incluirá tais mudanças no próximo commit efetuado ----- ## Versionando meus arquivos git status # exibe o status do repositório atual ----- ## Versionando meus arquivos git commit -m "adicionando o primeiro arquivo do meu projeto" ----- ## Verificando minhas alterações git log # exibe os commits efetuados ----- ## Utilizando repositórios de outras pessoas git clone https://github.com/fabrica-de-software-2018/amigos-frontend.git cd amigos-frontend Este link dá um ajuda: http://bit.ly/a-fabrica-2018 ----- ## Utilizando repositórios de outras pessoas git remote -v # lista todos os repositórios remotos aos quais este está relacionado ----- # Exercício! Mudar o título da página ----- ## Utilizando repositórios de outras pessoas * Após commitar, você precisa enviar as alterações para este usuário (ou servidor) remoto * desta forma você compartilha seu código com outras pessoas git push # "empurrar" suas alterações para o usuário remoto ----- ## Utilizando repositórios de outras pessoas * é importante que você também tenha acesso ao repositório * decidam na sorte quem vai ganhar acesso primeiro ----- # Agora todo mundo pode ter acesso :p ----- ## Utilizando repositórios de outras pessoas * Nem sempre dá certo, afinal outras pessoas podem ter efetuado alterações nesse meio tempo * é através desse mecanismo que o git garante que ao enviar (fazer o push) suas alterações, você não danifica a alteração de outrem * você precisa buscar as alterações novas do repositório: git pull # "puxar" alterações do usuário remoto ----- # Conflito! ----- ## Tá, não tou afim de resolver esse conflito agora, depois eu vejo Tem um comando que apenas baixa as alterações, mas não aplica elas no seu código: git fetch origin # lembra que o remoto se chama origin? git merge origin/master O `git pull`, na verdade, faz o comando de fetch (baixa alterações) e aplica (merge) no teu código ----- # Referências/Materiais de apoio: * https://services.github.com/on-demand/downloads/pt_BR/github-git-cheat-sheet.pdf * https://git-scm.com/book/pt-br/v2 * https://try.github.io * tig ----- # Perguntas? ----- # Perguntas! * e se eu quiser testar uma coisa diferente no código sem estragar os commits que fiz? * e se eu quiser apagar todas as mudanças locais? * e se eu quiser reverter o que fiz, quiser voltar atrás? * ... ----- # Agora podemos tentar colocar o projeto de vocês no git, o que vocês acham? ----- # Valeu!
